# Playground 每日创意冲刺流程

> 目的：把“23:00 创意冲刺”从提醒型任务，改成**默认由小小鱼主动执行的闭环任务**。
> 
> 默认时区以用户侧北京时间（UTC+8）为准。

## 一、任务定义

每日北京时间 23:00，启动一次 Playground 创意冲刺。

核心目标：
1. 先提出 3-5 个可玩、可做、偏轻量的 idea
2. 发给郑禹君，让他有 30 分钟选择窗口
3. 同时把 ideas 和时间戳记录到指定 Notion 页面
4. 到 23:30 做去重检查
5. 若尚未闭环，则继续执行，直到完成：
   - 选题确认（用户选 or 我自选）
   - coding
   - commit / push
   - Notion 记录

---

## 二、执行原则

### 1. 默认不是“提醒用户来做”
这个任务的执行主体是**小小鱼**。

除非需要用户做选择，否则不应该把任务原样转述成“提醒你该做了”。
正确做法是：
- 我先产出 ideas
- 我先发起选择
- 我先落 Notion
- 到点后我继续补完开发与记录

### 2. 用户只负责“选”
在 23:00 ~ 23:30 的窗口内：
- 用户如果回复了选择，我按用户选题执行
- 用户如果没回复，我就自己从当天 ideas 中选一个继续做

### 3. 只在需要时发消息
消息应该服务于闭环，而不是制造噪音。

**应发消息：**
- 23:00 发当天 3-5 个 ideas 给用户选
- 23:30 若用户没选，可发一句“我先替你选一个继续做了”
- 完成后发结果摘要（做了什么、repo/目录、可体验方式）

**不应发消息：**
- 单纯复述系统提醒
- 已经完成闭环后再重复催促
- 明知该我执行却把任务丢回给用户

---

## 三、23:00 阶段标准动作

北京时间 23:00 到达后，立即执行：

### Step 1. 生成 ideas
产出 3-5 个 idea，要求：
- 有趣
- 轻量
- 30~90 分钟内能做出一个最小可用版本
- 尽量适合网页、小工具、交互 demo、趣味实验
- 避免高风险、侵入性、灰色或有害方向

每个 idea 建议包含：
- 名称
- 一句话说明
- 为什么值得做
- 最小可交付物（MVP）是什么

### Step 2. 发给郑禹君选择
消息模板可参考：

```text
今晚的 Playground 创意冲刺我先想了 4 个方向，你可以在 30 分钟内选一个；如果你没回，我会自己挑一个继续做完。

1. xxx
2. xxx
3. xxx
4. xxx
```

要求：
- 明确说明“30 分钟内可选”
- 明确说明“若未回复，我会自行选择并继续完成”
- 语气简洁，不拖泥带水

### Step 3. 记录到 Notion
在指定 Notion 页面记录：
- 日期
- 时间戳（北京时间）
- 当天的 3-5 个 ideas
- 当前状态：`等待用户选择（23:00 发出）`

Notion 页面：
- <https://www.notion.so/Yu-Claw-31d3838add7a80979237e5d1640fabe0>

## 四、23:30 阶段去重与补全逻辑

北京时间 23:30 到达后，先做去重检查。

### A. 已完成闭环 → 直接结束
如果以下条件都满足，则**不再发任何提醒消息**，直接结束：
- 用户已选择，或已完成自选
- coding 已完成
- commit / push 已完成
- Notion 已更新最终结果

此时应输出：
- `NO_REPLY`

### B. 未完成 → 继续补全
如果尚未闭环，则分两种情况：

#### 情况 1：用户已回复选题
则按用户选择继续执行：
1. 开始 coding
2. 本地验证 / 最小试跑
3. commit
4. push
5. 更新 Notion：记录最终选题、结果、repo 路径、时间戳
6. 向用户发送完成摘要

#### 情况 2：用户未回复
则由我自行从当晚 ideas 中选一个最值得做的题：
1. 选题
2. 开始 coding
3. 本地验证 / 最小试跑
4. commit
5. push
6. 更新 Notion：注明“用户未在 30 分钟内选择，故由小小鱼自选执行”
7. 视情况给用户发一句简短说明，再发送完成摘要

## 五、开发执行标准

### 选题优先级
优先选择：
1. 能在当天完成最小闭环的
2. 有趣且容易展示的
3. 可积累成 Playground 作品集的
4. 技术风险低、反馈快的

### 实现要求
默认至少做到：
- 有实际代码产出
- 能运行或可静态预览
- 有简短 README 或项目说明
- 不留一个空壳目录冒充完成

### 项目目录建议
建议放在：
- `projects/idea_YYYY_MM_DD_<slug>/`

例如：
- `projects/idea_2026_03_17_micro_mood_mixer/`

### README 更新建议
完成后可在仓库 `README.md` 追加：
- 日期
- 今日选题
- 项目目录
- 一句话说明

## 七、Notion 记录规范

每次至少记录以下信息：

### 23:00 初始记录
- 日期
- 北京时间时间戳
- ideas 列表
- 状态：等待选择

### 最终完成记录
- 最终选题
- 选择来源：用户选择 / 小小鱼自选
- 项目路径
- commit 信息（可写摘要，不要泄漏敏感信息）
- 完成时间戳
- 试跑结果 / 可预览方式

## 八、消息模板

### 1. 23:00 发 ideas
```text
今晚的 Playground 创意冲刺我先整理了 4 个 idea，你可以在 30 分钟内选一个；如果 23:30 前你没回，我会自己挑一个继续做完。

1. Idea A：...
2. Idea B：...
3. Idea C：...
4. Idea D：...
```

### 2. 23:30 用户未回复
```text
你这边还没选，我就先替你挑一个往下做了。做完后我直接把结果发你。
```

### 3. 完成后汇报
```text
今晚这轮 Playground 已做完：
- 选题：...
- 路径：...
- 试跑结果：...
- 已 commit / push
- Notion 已更新
```

## 九、失败兜底

如果执行过程中遇到阻塞（如：
- Git 远端权限异常
- push 失败
- Notion API / 凭据不可用
- repo 状态冲突严重
），则遵循以下原则：

1. 先尽量完成本地可完成部分
2. 明确记录卡点
3. 向用户汇报“已完成部分 + 未完成原因 + 下一步建议”
4. 不要伪装成已经全部完成

---

## 十、完成定义（Definition of Done）

一次 Playground 每日创意冲刺，只有满足以下全部条件才算完成：

- [ ] 当天 ideas 已生成
- [ ] 已发给用户，并给出 30 分钟选择窗口
- [ ] 已写入 Notion 初始记录
- [ ] 已确定最终选题（用户选 / 我自选）
- [ ] 已完成 coding
- [ ] 已完成最小试跑验证
- [ ] 已 commit
- [ ] 已 push
- [ ] 已更新 Notion 最终结果
- [ ] 已向用户发送最终完成摘要

---
